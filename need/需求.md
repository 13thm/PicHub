一、项目整体概述
基于 SpringBoot + React 实现一套多图片协同编辑与实时预览系统，支持多用户同时在线、单图片同一时间仅一人可编辑、所有在线用户实时查看操作、系统消息实时广播，核心围绕多张图片独立进行权限控制、操作同步与消息通知。
二、角色与用户行为
普通在线用户
可查看任意一张图片
可查看图片实时编辑操作（旋转）
可接收对应图片的系统消息
当图片被占用编辑时，编辑按钮置灰不可用
编辑用户（同一图片仅一人）
可进入图片编辑模式
可对图片执行左旋、右旋操作
操作实时同步给所有查看该图片的用户
编辑完成可保存至后端
保存 / 异常退出后释放编辑权限
三、核心功能需求
1. 多图片管理
   系统内存在多张独立图片，每张图片拥有唯一标识（图片 ID）
   多张图片可同时被不同用户分别编辑，互不干扰
   每张图片独立维护：编辑状态、旋转角度、编辑者信息
2. 编辑互斥控制（核心）
   同一时间，一张图片只允许一个用户进入编辑模式
   用户点击「进入编辑」时，后端校验该图片是否已被占用
   未占用：允许进入，并锁定该图片，其他用户按钮禁用
   已占用：拒绝进入，前端提示 “当前图片正在编辑中”
   编辑用户关闭页面 / 断开连接，系统自动释放该图片编辑锁
   编辑状态全局实时同步，所有用户按钮状态保持一致
3. 图片编辑操作
   编辑模式下提供两个操作：
   左旋：图片向左旋转 90 度
   右旋：图片向右旋转 90 度
   旋转操作不立即保存到数据库，仅实时同步展示
   点击保存后，将最终旋转后的图片提交后端持久化
   保存成功后，自动退出编辑模式、释放图片编辑锁
4. 实时同步预览
   所有正在查看同一张图片的用户，可实时看到编辑者的旋转操作
   新进入查看某张图片的用户，直接看到该图片当前最新旋转状态
   旋转操作只同步给查看该图片的用户，不同图片之间不同步
5. 实时消息广播
   消息基于当前查看的图片进行广播，不跨图片推送
   需广播的消息类型：
   用户进入查看某张图片
   用户开始编辑某张图片
   用户对图片执行旋转操作（可选记录）
   用户保存图片完成编辑
   用户退出 / 异常断开导致编辑锁释放
   所有在线查看该图片的用户均可收到消息并展示在消息区
6. 连接与切换
   用户可切换查看不同图片
   切换时自动离开原图片频道，加入新图片频道
   不同频道的编辑状态、操作、消息完全隔离
   四、非功能需求
   实时性：旋转操作、消息、编辑状态延迟极低，接近实时
   隔离性：多图片、多用户之间互不干扰，不串状态、不串消息
   可靠性：编辑用户异常断开时，自动解锁，避免图片一直被占用
   一致性：所有用户看到的图片状态、编辑按钮状态完全一致
   五、核心技术实现依据（简要）
   使用 WebSocket 实现前后端长连接与实时双向通信
   使用 Channel（频道 / 房间）机制 按图片 ID 隔离用户
   后端使用 Map<图片ID, 编辑状态> 实现多图片独立编辑锁
   后端使用 Map<图片ID, 在线用户集合> 实现按频道消息推送


后端实现基本步骤（最终精简版）
1. 搭建基础环境
   引入核心依赖：
   WebSocket（必须）
   图片处理工具（如 thumbnailator）
   存储（本地 / OSS / 数据库）
2. 设计 3 个核心全局状态（后端内存维护）
   这是多图片协同的灵魂：
   图片编辑锁
   Map<图片ID, 编辑信息(是否编辑中、编辑者ID、sessionId)>
   WebSocket 频道 / 房间管理
   Map<图片ID, 连接会话列表>
   图片实时旋转角度
   Map<图片ID, 当前旋转角度>
3. 实现 WebSocket 核心配置
   配置 WebSocket 端点、处理器、拦截器
   实现连接创建、断开、消息接收三大事件
4. 实现 “频道机制”（多图片隔离核心）
   给每个图片做独立房间：
   加入房间：用户查看某张图 → 加入对应图片频道
   离开房间：用户切换 / 退出 → 离开旧频道
   广播：只发给当前频道内的用户
5. 实现编辑互斥锁逻辑（核心）
   请求编辑 → 检查该图片是否被锁定
   未锁定 → 上锁，记录编辑者，广播编辑状态
   已锁定 → 拒绝
   断开连接 / 退出编辑 → 自动解锁
   所有状态通过 WebSocket 实时推送到全频道
6. 实现图片旋转实时同步
   编辑者发送：图片 ID + 旋转方向（左 / 右 90 度）
   后端更新该图片的当前角度
   后端广播给该频道所有用户
   前端本地渲染旋转（不存图）
7. 实现图片保存接口
   提供 HTTP 接口：接收图片 ID + 最终图片文件
   后端覆盖更新图片（持久化）
   释放编辑锁
   广播 “保存成功” 消息到频道
8. 实现消息广播工具
   统一方法：
   按频道广播
   支持消息类型：进入、离开、开始编辑、旋转、保存、解锁
9. 处理异常断开（关键）
   用户关闭页面 → WebSocket 断开：
   判断断开的是不是当前图片的编辑者
   如果是 → 自动解锁
   广播：编辑异常退出
10. 提供基础查询接口
    获取所有图片列表
    获取单张图片详情 + 当前状态
    新用户进入时同步最新状态

四、必须提供的 后端接口（2 个即可）
都是 HTTP 接口，你原有项目可直接兼容
1. 图片保存接口（必选）
   POST /api/image/edit/save
   参数：图片 ID + 旋转后的图片文件
   功能：保存图片 → 解锁 → 广播成功
2. 获取图片当前状态接口（可选）
   GET /api/image/{imageId}/status
   返回：是否编辑中、当前角度、编辑者
   五、前端通过 WebSocket 会调用的 指令类型

   join 加入图片频道
   leave 离开频道
   edit 请求编辑图片
   rotate 旋转图片（左 / 右）
   save 保存并退出编辑
   六、后端需要 广播给前端的消息类型
   用户进入
   用户离开
   开始编辑
   旋转同步
